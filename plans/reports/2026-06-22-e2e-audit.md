# learn-everything — end-to-end audit (method · effect · arrangement)

Date: 2026-06-22
Scope: full learning flow — pedagogy (method), understanding+retention (effect), scheduling/orchestration/UX (arrangement).
Inputs: scripts/fsrs.py, scripts/registry.py, methods/*.md, skills/learn/SKILL.md, plans/specs/*, tests/*.
All findings below were adversarially verified against the real files (line cites reproduce verbatim). The real
`tracks/datawhale-llm` data was not modified; CLI probes used throwaway ids and were cleaned up.

## Summary

The product's stated architecture holds up: pip-free stdlib engine, markdown-as-source-of-truth, Obsidian-compatible
cards, skill-first single adapter. But three systemic gaps run through the whole flow:

1. **The default pedagogy (`tutor`) is invisible to the orchestrator.** The host model reads enum/defaults that omit
   `tutor`, even though it is the declared default and the only real track ships it. New domain tracks silently default
   to `socratic` against the read-along intent.
2. **The "do this next" surface is the weakest link.** `status_board()` is the single shipped cross-track surface, yet it
   discards most of the fields the model needs (next_action, mode, pedagogy, cards_total, stale magnitude) and returns an
   alphabetical, unranked list — so prioritization, the product's headline differentiator, is delegated to an LLM
   vibe-guess.
3. **The retention safety net is never deployed, and is unsafe when it is.** Nothing nudges a card-less track (the one
   real track has a full tutor session and 0 cards), card approval can't batch, there is no grade history for leech
   detection, and grading on a corrupt/malformed review-state silently wipes other cards' FSRS state.

The pedagogy is articulate but partly inert: `learning-science.md` mandates MISSION.md / glossary.md / learning-records/
that no code creates and no step fires; every method targets the struggling learner with no up-shift for the bored one;
two files grade the same Feynman teach-back differently. None of these break the flow, but they hollow out the "teaching-
first" claim in practice.

Counts: 11 quick fixes (applyable now), 10 design changes (need a spec decision or engine work).

## Per-dimension notes

### Method (pedagogy)
- The strongest single fix is Q3: a default pedagogy the orchestrator can't see is a coherence bug at the front door —
  it makes every new domain track default to question-first against the read-along product.
- The method files are uniformly built for the struggling learner (tutor.md:29 down-shift only; socratic tiered hints;
  feynman gap-probing). There is no executing up-shift move (Q5/D6) despite the ZPD principle at learning-science.md:55.
- Grading is double-specified (Q4): feynman's 2-bucket "explanation cleanliness" vs active-recall's 1-4 retrieval-by-feel
  — and active-recall.md:84 explicitly bans grading on understanding-after-reveal, which is what feynman.md does.
- Card-quality lever (L1/L2/L3 layering, Q6) lives only in the review-method file and is never referenced at card birth.

### Effect (understanding + retention)
- The retention net is real in the engine but undeployed in practice (D4): the one real track teaches and retains nothing
  (0 cards after a full session). Surface card-coverage so the model nudges distillation.
- Leech handling is a policy with no mechanism (D5): "keeps failing → re-teach" has no defined trigger and no grade
  history because grade_card overwrites per-card state. The feature-designs review-log is already planned — ship it.
- Retention state is fragile: corrupt review-state silently wipes other cards (Q8); a single malformed entry crashes the
  default STATUS command (Q9); non-atomic writes risk corruption under Drive sync (D11). These compound: D11 produces the
  corruption that Q8 then amplifies.
- learning-science.md is a well-specified discipline with no firing hook (D3): MISSION/glossary/learning-records are
  preached but never created — "sounds great, does nothing."

### Arrangement (scheduling / orchestration / UX)
- status_board() is the product's only shipped cross-track surface and it both under-reports (Q1) and mis-orders (Q2/D1).
  These two fixes give the host model an actually-actionable, deterministically-ranked board for near-zero cost.
- Card approval doesn't scale (D2): the single human "approve these N" gate degrades into N subprocess calls with a
  partial-write hazard, contradicting the spec's own one-batch promise.
- Resume degrades silently when `--next` was skipped (D7); the live-MOC never knits (D8); staleness is detected but inert
  (D9). All three are "works for the toy case, frays on the real long-document curriculum" arrangement gaps.
- INGEST mixes engine-owned CLI calls and model-owned file writes with no marking (Q7), blurring the
  engine-owns-state/model-owns-pedagogy contract exactly where both happen.

## Suggested apply order
1. Engine safety first: Q8, Q9, Q10, Q11 (data-integrity; small, isolated).
2. Board leverage: Q1 + Q2 in one edit (highest UX leverage for the cost).
3. Pedagogy coherence: Q3, then Q4, Q5, Q6, Q7.
4. Backlog: D1–D5 are the high-impact design work (ranker, batch approval, scaffolding reachability, retention nudge,
   review-log); D6–D11 follow.
